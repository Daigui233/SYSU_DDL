# Slave_TC264D 下位机说明

## 一、工程定位

本目录为 `SYSU_DDL` 项目的下位机工程，运行平台为逐飞 `TC264D` 开源库。

下位机在整套系统中的职责是：

- 接收上位机发送的控制相关信息
- 完成电机、舵机等执行机构驱动
- 完成编码器测速与底层闭环控制
- 承担状态执行、安全停车、基础控制等任务

当前阶段的主要原则是：

- 先把下位机基础框架搭稳
- 先把模块边界划清楚
- 先保证底层驱动与控制链路可扩展
- 不为了当前单一赛道任务把后续架构写死

## 二、当前任务目标

当前第一阶段目标为：

- 先完成椭圆环赛道稳定跑动
- 先完成电机、舵机、PID、初始化、通信等基础模块
- 在此基础上，为后续正式任务预留扩展空间

当前不是只做一份临时巡线代码，而是要搭出一套后续能继续承接正式赛题的下位机框架。

## 三、后续赛题扩展方向

根据当前讨论与规则理解，后续正式任务将逐步包含以下内容：

- 行人避让
- OCR 路牌识别与语义导航
- 红绿灯识别与通行控制
- 限速标志识别与速度约束
- 金币收集与路径收益权衡

因此，下位机当前虽然先服务于椭圆环跑动，但在设计时必须提前考虑：

- 控制链路是否可扩展
- 通信协议是否可扩展
- 状态管理是否可扩展

## 四、下位机职责分层规划

当前建议将下位机分为以下几层：

### 1. 执行层

直接驱动底层硬件：

- 电机 PWM 输出
- 舵机 PWM 输出
- 编码器测速
- 必要的停止、保护与安全输出

当前已开始搭建：

- `Motor`
- 后续 `Servo`

### 2. 控制层

负责闭环控制算法：

- 电机速度环 PID
- 舵机转向 PD / PID

控制层只关心：

- 目标值
- 反馈值
- 控制输出

控制层不直接关心 OCR、行人、红绿灯等语义任务。

### 3. 行为层

负责把上位机感知结果转换为可执行控制目标，例如：

- 正常巡线
- 减速巡航
- 限速通行
- 红灯等待
- 避障绕行
- 岔路转向

行为层最终输出建议统一为：

- 目标速度
- 目标转向
- 控制模式
- 急停标志

### 4. 状态层

负责当前任务上下文管理，例如：

- 当前圈数
- 当前所处赛道区域
- 当前是否处于限速任务
- 当前是否处于红灯等待
- 当前是否处于避障接管
- 当前是否执行金币策略

当前阶段状态机只保留最小必要能力，后续再逐步丰富。

## 五、当前控制链路规划

当前建议采用如下控制链路：

1. 上位机输出赛道误差、目标速度与场景信息
2. 下位机通信层完成串口收包与校验
3. 下位机行为层根据当前场景生成控制目标
4. 电机控制器根据目标速度与编码器测速输出 PWM
5. 舵机控制器根据目标转向误差输出舵机 PWM
6. 下位机将必要状态回传给上位机

这一链路的目标是把：

- 感知
- 决策
- 执行

三部分职责清晰分离，避免后续任务接入时频繁推翻底层代码。

## 六、当前控制方式规划

### 1. 电机控制

电机采用以下方式：

- 编码器测速反馈
- 双 PWM 正反转输出
- 固定周期速度闭环

当前建议：

- 固定周期读取编码器速度
- 用 PID 完成速度控制
- 对电机占空比进行限幅

当前 `Motor` 模块中已将编码器与电机执行暂时放在同一模块内，原因是：

- 当前编码器主要服务于电机速度闭环
- 暂时没有必要单独拆分出 `Encoder` 模块

若后续出现以下需求，再考虑独立拆分编码器模块：

- 里程估计
- 距离控制
- 多模块共用编码器数据
- 左右轮独立测速与建模

### 2. 舵机控制

舵机采用以下方式：

- 独立模块
- 固定频率 PWM 输出
- 中值与左右阈值单独标定

当前建议：

- 先采用 PD 控制
- 后续视赛道表现再决定是否引入积分项

## 七、模块搭建顺序规划

当前建议按以下顺序逐个搭建模块：

1. `Motor`
2. `Servo`
3. `PID`
4. `Init`
5. `Communication`
6. `Control`
7. `State`

当前进度：

- `Motor` 第一版已完成并审查通过
- `Servo` 待开始
- 其余模块后续逐步搭建

## 八、通信协议设计原则

当前上下位机通信协议虽然尚未最终定稿，但已经明确以下原则：

- 保留旧协议“帧头 + 地址 + 长度 + 数据 + 校验”的组织形式
- 不直接沿用旧赛题中的状态语义
- 不能只围绕当前椭圆环任务设计
- 必须为 OCR、避障、红绿灯、限速、金币等内容预留字段

也就是说，当前通信协议不仅要能服务于巡线，还要能支撑未来更复杂的感知结果下发与控制接管。

## 九、暂定上位机 -> 下位机协议草案

当前讨论中的下行控制帧建议至少包含以下信息：

- 帧头 `Head`
- 功能地址 `Addr`
- 帧长度 `Length`
- 赛道横向误差 `track_error`
- 赛道角度误差 `track_angle_error`
- 目标速度 `target_speed`
- 场景类型 `scene_type`
- 标志位 `flags`
- 导航方向 `nav_dir`
- 限速值 `speed_limit`
- 障碍距离 `obstacle_distance`
- 校验和 `checksum`

### 建议预留的场景类型

- `SCENE_TRACK`
- `SCENE_AVOID_PEDESTRIAN`
- `SCENE_NAVIGATION_SIGN`
- `SCENE_TRAFFIC_LIGHT`
- `SCENE_SPEED_LIMIT`
- `SCENE_COIN`

### 建议预留的标志位

- 急停标志
- 行人有效标志
- 红灯标志
- 限速有效标志
- 金币有效标志
- 导航结果有效标志
- 障碍物有效标志

## 十、下位机 -> 上位机反馈建议

下位机回传建议至少包含：

- 当前实际速度
- 当前编码器速度
- 当前电机输出
- 当前舵机输出
- 当前控制模式
- 当前状态机状态
- 当前圈数
- 错误码

这样上位机后续不仅能做调试，也能在必要时感知下位机执行状态。

## 十一、当前代码规范约定

为了保证多人协作一致性，当前约定如下：

- 下位机源码统一保持 `GBK` 编码，以匹配逐飞库
- 模块代码注释风格尽量参考逐飞库
- 文件头信息保持统一格式
- 每个模块完成后先审查，再继续下一个模块
- 每次提交尽量围绕单一模块或单一功能展开

## 十二、当前下位机开发约定

当前阶段有以下明确约定：

- 先做 `Motor`，再做 `Servo`
- 再逐步推进 `PID`、`Init`、`Communication`
- 所有模块优先保证清晰可审查
- 底层驱动先稳定，再叠加行为与状态逻辑

## 十三、协作方式建议

为了便于队员共享开发，建议采用以下协作方式：

- 一个模块一个分支
- 一个功能一个提交
- 先本地测试，再合并主分支
- 所有接口和注释保持统一风格
